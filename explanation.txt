3.1 Node.js —— 为什么用它？
Node.js 是什么：一种让 JavaScript（平时只在浏览器里跑的脚本语言）也能在服务器上运行的技术。它最大的特点是「不等人」——发出去一个任务后，不等结果回来就去处理下一个任务了（这叫非阻塞 IO）。

       → 适合每个人的单子很简单，但人特别多的情况

专业术语对照：

非阻塞 IO = 不等结果，边等边做别的事
事件驱动 = 结果回来了会「通知」我，而不是我一直盯着看
单线程 = 一个人干活，但效率很高因为不傻等

为什么要两份？

类比：

AES 加密 = 带锁的日记本。有钥匙的人能翻开看内容
SHA256 哈希 = 指纹识别。能认出「哦这是同一个人」但无法从指纹反推出长相
专业术语对照：

AES-256-CBC = 一种被全球银行和政府广泛使用的加密方法。256 表示密钥长度 256 位（极难暴力破解），CBC 表示一种链接模式让密文更安全
SHA256 = 一种哈希算法。输入任意文字，输出固定 64 个字符的「指纹」。哪怕输入只改了一个标点，输出也完全不同
加盐（Salt） = 哈希之前混入一段额外的文字，防止别人用「彩虹表」（预先算好的常见密码→哈希对照表）反推出原文
加密 vs 哈希：加密是可逆的（能解密还原），哈希是不可逆的（像把一张纸烧成灰，无法从灰还原成纸）

专业术语对照：

JWT（Json Web Token） = 三部分拼起来的字符串：头部.荷载.签名。类似「护照信息 + 防伪水印」
RS256（非对称签名） = 有一对密钥：私钥（只有自己知道，用来盖章）和公钥（可以公开，用来验章）。即使公钥泄露，没有私钥也造不出合法的签名
HttpOnly（Cookie 安全标志） = 告诉浏览器「这个 Cookie 只能自动带着访问服务器，不允许网页里的 JavaScript 程序读取」
JTI（JWT ID） = 每张门禁卡的唯一编号。管理员要踢人下线，就把这个编号加入黑名单，卡就失效了

核心思路：服务器生成一个随机的正确位置，存起来；用户滑过去后比对偏差。

生成验证码：
1. 服务器随机决定：「拼图块应该在横向第 150 像素的位置」
2. 把这个「150」存进 Redis，设置 60 秒后自动删除
3. 返回给手机：「这是一张背景图+一个拼图块，拼图块的缺口在(150, 50)」

用户滑动验证：
1. 用户把滑块拖到某个位置（比如 148 像素）
2. 手机把位置「148」发给服务器
3. 服务器从 Redis 取出正确答案「150」
4. 偏差 = |148 - 150| = 2 像素 ✅（容差 ±5 像素）
5. ⚠️ 立即删除这个答案！无论对错都删！
6. 通过 → 发一个一次性 token（5 分钟有效）

十一、面试可能问的问题（用人话回答）
Q1：为什么选 Node.js 而不是 Java？
回答：风控系统的瓶颈不在 CPU 算力，而在「等网络返回」。每次注册要等 Redis 和 MySQL 的响应，Node.js 在等的时候就去处理下一个请求了，效率很高。实际测下来单机每秒能处理 1300 多次注册。Java 当然也能做，但对于这类 IO 密集型任务，Node.js 代码更少、启动更快、Docker 镜像也更小。

Q2：Redis 和 MySQL 怎么分工的？
回答：Redis 快但数据会丢（重启就没了），MySQL 慢但永久保存。所以用了「快慢结合」——Redis 做实时拦截（黑名单 O(1) 查找、计数器），MySQL 做永久存档和审计。注册时先查 Redis，命中了直接拒绝（亚毫秒），没命中再查 MySQL 兜底。

Q3：如果有人拿到了你们的数据库，手机号会泄露吗？
回答：不会。手机号在数据库里存的是加密乱码（AES-256-CBC）。每条数据用不同的随机 IV 加密，即使两条相同的手机号也会产生完全不同的密文。要解密必须要有密钥，而密钥只在服务器的环境变量里，不在数据库里。另外，黑名单匹配用的是 SHA256 哈希（不可逆），全程不需要解密。

Q4：JWT 不是有「没法中途作废」的缺陷吗？你们怎么解决的？
回答：对，这是 JWT 的经典问题——签发出去了，在到期之前就一直有效。我们通过「JTI 吊销黑名单」来解决：每个 JWT 签发的时给一个唯一编号（JTI），要踢人下线时把这个编号写入 Redis 黑名单。之后每个请求先查黑名单（O(1)，不费时间），命中了就拒绝。这样就相当于给门禁卡加了「挂失」功能。

Q5：分布式场景下会有什么问题？
回答：目前是单机部署，但如果有多个服务器实例：

限流计数器会出现问题：同一个 IP 的请求被分发到两台机器上，每台各自计数，加起来可能超过阈值。解决方法是把计数器统一放在 Redis 里（Redis 是独立进程，多台机器共享同一个 Redis）。
要解决「多个服务器同时修改 Redis 同一个 key」的竞态问题，需要加分布式锁（分布式锁 = 一个标记，谁拿到了谁操作，其他人排队等着）。
Q6：验证码答案为什么「不管对错都删」？
回答：防止暴力破解。如果不删除，攻击者拿到一个 captchaId 后可以反复试探（猜 x=1、x=2、x=3...），总能猜对。删了之后，一次尝试就消耗一个答案，要再试必须重新获取验证码，而获取验证码又被全局限流控制，攻击成本大幅增加。

Q7：项目有什么可以改进的地方？
回答：

注销时的多步 MySQL 操作没加事务（原子性保证），极端情况下可能出现数据不一致
目前是单节点部署，要扩展到多节点需要加分布式锁
风控目前只看手机号 + IP + 设备 ID，可以加入用户行为分析（滑动轨迹、点击速度、页面停留时间等）
拦截日志只存在 MySQL 里查询不方便，可以接入 Grafana 做可视化实时监控大屏

  类比：你的银行卡只能在你的银行的系统里用

→ 防御：JTI + Redis 黑名单，可以远程挂失
  类比：酒店前台可以注销丢失的房卡
